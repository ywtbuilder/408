数据很多时，想让**查找、插入、删除**都尽量快，核心就一句话：

> **不要靠“从头到尾一个个找”，要用合适的数据结构给数据建立“索引”。**

比如你有 100 万个人的信息，如果每次都从第 1 个看到第 100 万个，这叫**线性查找**，很慢，时间复杂度是 `O(n)`。

更好的办法是：根据“身份证号 / 学号 / 用户 ID / 文件名”这种唯一标识，直接定位。

---

## 1. 最常用答案：哈希表 Hash Table

如果你的需求是：

> 根据某个 key 找数据，比如根据学号找学生、根据手机号找用户、根据单词找解释

那最适合的是：

```text
Hash Table / HashMap / Dictionary / Map
```

它的平均复杂度是：

| 操作 |  平均复杂度 |
| -- | -----: |
| 查找 | `O(1)` |
| 插入 | `O(1)` |
| 删除 | `O(1)` |

也就是数据很多时，仍然很快。

比如：

```text
学号 1001 -> 张三
学号 1002 -> 李四
学号 1003 -> 王五
```

查 `1002` 的时候，不用从头找，哈希表会根据 `1002` 算出它大概放在哪个位置，然后直接去拿。

可以理解成：

```text
普通数组/链表：像在一堆书里一本本翻
哈希表：像图书馆有编号，直接去对应书架
```

---

## 2. 但是哈希表不是万能的

哈希表适合：

```text
精确查找
```

比如：

```text
找 id = 10086 的用户
找 name = "Tom" 的记录
删除 key = "apple" 的单词
```

但是它不擅长：

```text
范围查询
排序查询
找最大值/最小值
找第 10 到第 20 个数据
```

比如你想查：

```text
分数在 80 到 90 之间的学生
价格从低到高排序
找比 100 大的第一个数
```

这时哈希表就不合适了。

---

## 3. 如果需要有序：用平衡二叉搜索树

如果你既要查找快，又要保持数据有顺序，就用：

```text
Balanced Binary Search Tree
平衡二叉搜索树
```

常见实现有：

```text
AVL Tree
Red-Black Tree 红黑树
TreeMap
TreeSet
```

它的复杂度一般是：

| 操作 |        复杂度 |
| -- | ---------: |
| 查找 | `O(log n)` |
| 插入 | `O(log n)` |
| 删除 | `O(log n)` |

`O(log n)` 虽然不如哈希表的平均 `O(1)`，但也非常快。

举个感觉：

```text
100 万条数据
O(n)：最多找 100 万次
O(log n)：大概找 20 次左右
```

所以平衡树很适合：

```text
需要排序
需要范围查询
需要找前驱/后继
需要动态插入删除
```

---

## 4. 如果是数据库、大文件：用 B 树 / B+ 树索引

如果数据多到存在数据库、磁盘里，比如 MySQL、SQLite、文件系统，就常用：

```text
B-Tree
B+Tree
```

数据库里的索引本质上就是这个思路。

比如：

```sql
CREATE INDEX idx_user_id ON users(user_id);
```

这句话的意思就是：

> 给 user_id 建一个索引，以后根据 user_id 查找就不用全表扫描。

没有索引时：

```text
从第一行扫到最后一行
```

有索引后：

```text
像查字典目录一样快速定位
```

所以数据库里数据很多时，最重要的优化就是：

```text
给经常查询的字段建立索引
```

---

## 5. 数组、链表、哈希表、树的对比

| 数据结构                   |                   查找 |          插入 |          删除 | 适合场景      |
| ---------------------- | -------------------: | ----------: | ----------: | --------- |
| 数组 Array               | 按下标 `O(1)`，按值 `O(n)` | 中间插入 `O(n)` | 中间删除 `O(n)` | 按位置访问     |
| 链表 Linked List         |               `O(n)` | 已知位置 `O(1)` | 已知位置 `O(1)` | 频繁插删但不常查找 |
| 双链表 Doubly Linked List |               `O(n)` | 已知节点 `O(1)` | 已知节点 `O(1)` | 前后移动方便    |
| 哈希表 HashMap            |            平均 `O(1)` |   平均 `O(1)` |   平均 `O(1)` | 精确查找最快    |
| 平衡树 TreeMap            |           `O(log n)` |  `O(log n)` |  `O(log n)` | 有序、范围查询   |
| B+ 树 B+Tree            |           `O(log n)` |  `O(log n)` |  `O(log n)` | 数据库索引     |

---

## 6. 最实用的选择口诀

你可以这样记：

```text
只按 key 精确查找：HashMap
要排序、范围查询：TreeMap / 红黑树 / B+树
只是尾部追加：数组 / ArrayList
频繁从中间插删，而且已经知道位置：链表
数据库数据很多：建索引
```

---

## 7. 一个高级但很常见的组合：哈希表 + 双链表

有时候单独一种结构不够，会组合使用。

比如经典的 LRU 缓存：

```text
HashMap + Doubly Linked List
```

作用是：

```text
HashMap：负责快速查找，O(1)
双链表：负责快速移动和删除节点，O(1)
```

这就是为什么你刚学双链表时，经常会看到它和哈希表一起出现。

单独双链表：

```text
查找慢 O(n)
删除快 O(1)，前提是你已经拿到节点
```

加上 HashMap 后：

```text
先用 HashMap 快速找到节点
再用双链表快速删除/移动
```

于是查找、插入、删除都可以接近 `O(1)`。

---

## 最终结论

数据很多时：

```text
不要用普通遍历。
要根据访问方式选数据结构。
```

最常见选择是：

```text
精确查找：HashMap
有序查找：TreeMap / 红黑树
数据库大量数据：B+树索引
查找 + 快速删除移动：HashMap + 双链表
```

你可以把它理解成：

> 数据结构的本质，就是给数据设计一种“更快找到它”的摆放方式。
